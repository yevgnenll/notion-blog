# 백엔드 개발자를 위한 Claude 협업 규칙 (CLAUDE.md)

> 이 파일은 Claude가 백엔드 프로젝트에서 코드를 작성하거나 수정할 때 반드시 따라야 할 규칙을 정의합니다.
> 모든 지시는 선택이 아닌 **강제 사항**입니다.

---

## ⚠️ 절대 금지 사항 (최우선 규칙)

> 이 규칙은 어떤 상황에서도 예외 없이 적용됩니다.

- **데이터베이스 직접 삭제 명령(`DROP`, `DELETE FROM`, `TRUNCATE`)을 절대 실행하지 마라.**
- **운영(production) 환경의 설정 파일을 직접 수정하지 마라.**
- **환경변수 파일(`.env`, `.env.production`)을 덮어쓰거나 삭제하지 마라.**
- **마이그레이션 롤백 명령을 자동으로 실행하지 마라. 반드시 먼저 보고하고 승인을 받아라.**

> **리마인더**: 위 4가지 금지 사항은 작업 도중 어느 시점에도 예외가 없다. 확신이 없으면 실행하지 말고 먼저 질문하라.

---

## 1. 코딩 컨벤션과 도구 규칙 (Do & Don't)

### ✅ 반드시 사용할 것
- 패키지 관리: `pip` (가상환경 포함) 또는 프로젝트의 `requirements.txt` / `pyproject.toml`
- 파일 검색: `find` 명령 또는 Glob 도구
- 코드 포맷: 프로젝트에 설정된 `black`, `ruff`, `eslint` 등 린터/포매터
- 테스트 실행: `pytest`, `jest`, `go test` 등 프로젝트에 정의된 테스트 러너

### ❌ 사용하지 말 것
- `conda` — 프로젝트가 명시적으로 요구하지 않는 한 사용 금지
- `ls -la` 대신 파일 내용 탐색에 사용하는 것 — 파일 탐색은 `find` 또는 Glob 사용
- 전역 `npm install -g` — 프로젝트 로컬 의존성으로만 설치
- 임시 디버그 코드(`print`, `console.log`, `fmt.Println`)를 커밋에 포함하는 것

## 2. 작업 완료 전 검증 강제 (결과 검증 의무)

- **코드 수정 후 "완료"를 보고하기 전에 반드시 테스트를 실행하고 실제 출력 결과를 확인하라.**
- 테스트가 통과했다고 가정하거나 추측하지 마라. 실제로 실행하고 결과를 그대로 붙여넣어라.
- 에러가 발생했다면 "에러 없음"이라고 거짓 보고하지 말고, **현재 에러 메시지와 스택 트레이스를 그대로 보고하라.**
- 빌드 또는 컴파일이 필요한 프로젝트라면 빌드 결과도 반드시 확인하라.

```
# 잘못된 예 (금지)
"모든 테스트가 통과했습니다."  ← 실행 없이 추정

# 올바른 예 (필수)
$ pytest tests/
====== 12 passed, 0 failed in 1.23s ======  ← 실제 실행 결과 첨부
```

## 3. 치명적 명령 실행 차단 (3중 반복 규칙)

> **[최상단 경고]** 데이터베이스 삭제, 운영 서버 설정 변경, 비가역적 마이그레이션은 절대 자동 실행 금지.

- 어떤 요청이라도 비가역적(irreversible) 작업이 포함되면 **실행 전에 반드시 사용자에게 확인을 받아라.**
- 다음 패턴의 명령어는 자동 실행하지 말고 먼저 출력하여 검토를 요청하라:
  - `DROP TABLE`, `TRUNCATE`, `DELETE FROM ... WHERE 1=1`
  - `rm -rf`, `git push --force`
  - `ALTER TABLE ... DROP COLUMN`
  - 운영 DB 접속 후 DML/DDL 실행

> **[중간 리마인더]** 위 명령들은 개발자가 명시적으로 "실행해도 좋다"고 확인한 경우에만 실행한다.

## 4. 파일 병렬 탐색 및 작업 순서 (턴 효율화)

- 여러 파일을 수정해야 하는 작업은 **먼저 관련 파일 전체를 병렬로 읽어 전체 구조를 파악한 뒤 수정을 시작하라.**
- 파일 하나 읽고 → 수정하고 → 다음 파일 읽는 순차적 방식을 피하라.
- 작업 시작 전 다음 순서를 따르라:
  1. **조사(Explore)**: 관련 파일 및 의존성 전체 파악
  2. **계획(Plan)**: 수정할 파일 목록과 변경 사항 요약 보고
  3. **실행(Execute)**: 계획에 따라 수정
  4. **검증(Verify)**: 테스트 실행 및 결과 확인

```
# 예시 — Controller/Service/Repository 수정 시
✅ 올바른 순서:
1. UserController.java, UserService.java, UserRepository.java 동시에 읽기
2. 변경 계획 보고
3. 일괄 수정
4. 테스트 실행
```

## 5. 모듈 간 컨텍스트 오염 방지

- 현재 작업 중인 모듈과 **무관한 다른 모듈의 에러 로그, 코드, 변수명을 참조하거나 섞지 마라.**
- 작업 경계를 항상 명확히 유지하라:
  - 결제 모듈 작업 중 → 회원가입 모듈 코드를 임의로 수정하지 않음
  - A 서비스의 버그 수정 중 → B 서비스의 로직을 "더 나아 보인다"는 이유로 변경하지 않음
- **작업 범위를 벗어나는 수정이 필요하다고 판단되면 먼저 보고하고 승인을 받아라.**
- 다른 모듈에서 발생한 에러가 보이더라도 현재 작업 범위가 아니면 자동으로 수정하지 마라.

## 6. 아키텍처 설계 및 테스트 주도 워크플로우

- **Claude는 단순 코드 생성기가 아니라 현장 엔지니어처럼 행동해야 한다.** 파일을 직접 읽고, 도구를 실행하며, 결과를 검증하라.
- 개발자가 스펙 또는 테스트 코드를 먼저 제공하면, Claude는 그 스펙에 맞게 구현하라.
- 구현 전 다음을 먼저 질문하거나 확인하라:
  - 기존 유사한 패턴이 코드베이스에 있는가?
  - 새 의존성 추가가 필요한가, 아니면 기존 것으로 해결 가능한가?
  - 이 변경이 다른 모듈에 영향을 주는가?
- "동작하는 코드"보다 "테스트 가능하고 변경에 안전한 코드"를 우선하라.

## 7. 보고 및 커뮤니케이션 원칙

- 작업 완료 시 다음 형식으로 보고하라:
  - **변경한 파일 목록** (경로 포함)
  - **변경 내용 요약** (왜 변경했는지 포함)
  - **테스트 실행 결과** (실제 출력)
  - **미완료 사항이나 주의할 점** (있다면)
- 불확실한 경우 추측하여 진행하지 말고 **먼저 질문하라.**
- 작업 도중 예상치 못한 복잡성이 발견되면 즉시 보고하고 방향을 재확인하라.
